## New findings from validation run

### I-B3 (NEW IMPORTANT) — Cartesia voice_id namespace mismatch

**Source:** Validation re-run Cartesia 400 Bad Request.

**Description:** With I-B1 routing fix in place, the pipeline correctly attempted Cartesia for Korean + key set. But the Cartesia API call returned **400 Client Error** at `https://api.cartesia.ai/tts/bytes`. Voice ID passed was `pNInz6ob...` (an ElevenLabs voice ID — "Adam" — auto-assigned earlier when character.voice_id was empty).

Cartesia uses a different voice_id namespace from ElevenLabs. Passing an ElevenLabs voice ID to Cartesia → 400 (Cartesia doesn't recognize the ID).

**Fix scope (estimated):** ~10-30 LoC in `audio/dialogue.py`:
- Detect when provider is `CARTESIA_SONIC_2` AND character voice_id is empty OR appears to be an ElevenLabs ID (heuristic: 20-char alphanumeric matching ElevenLabs format)
- Substitute a Cartesia default voice (need to know one Cartesia voice_id for Korean — could use a hard-coded fallback OR query Cartesia API for the first Korean voice)
- Alternative: extend character schema to support `cartesia_voice_id` field separate from `voice_id`

**Disposition recommendation** (Rule #15 advisory matrix, IMPORTANT severity):
- **(a) Per-character Cartesia voice_id field**: extend Character schema with `cartesia_voice_id`. Backward compat preserved. Needs schema migration. ~20-40 LoC.
- **(b) Default Cartesia voice lookup**: add `_default_cartesia_voice_for_language(lang)` helper returning a known-good Cartesia voice_id per language. Quick fix; doesn't need schema change. ~15 LoC.
- **(c) NO ACTION**: graceful fallback to ElevenLabs is operating correctly; Cartesia routing demonstrably WORKS (per Run 3 attempt). For Tier C/D, accept ElevenLabs as the de-facto TTS provider; Cartesia validation deferred to cycle-16+ when voice_id namespace handling is fixed.

**Operator preferred:** **(c) NO ACTION** for Tier C — fallback works; cycle-16+ Cartesia voice_id work is its own concern. Pipeline correctness maintained.

### M-B3 (NEW MINOR) — amix duration=first clamps audio to dialogue length

**Source:** Validation re-run — ffprobe shows audio (3.9s) < video (5.1s).

**Description:** With C-B2 fix, the FFmpeg filtergraph uses `amix=inputs={N}:duration=first:dropout_transition=2`. When voice is a standalone short dialogue track (e.g., 3.9s for a 1-line dialogue), and video is longer (5.1s Kling Native default), `duration=first` clamps the output audio to dialogue length. Final video has 1.2s of silent tail.

For embedded-audio legacy path (Omnihuman/Veo), voice duration == video duration, so `duration=first` worked correctly. With standalone-dialogue path (my C-B2 fix), the mode should change.

**Fix scope:** 1-LoC change — `duration=first` → `duration=longest` when `use_standalone_dialogue` is True. With dialogue ≤ video, longest mode pads dialogue silence + plays BGM/foley to video end. Matches user expectation.

**Disposition recommendation** (Rule #15 advisory matrix, MINOR severity):
- **(a) Inline fold**: 1-LoC fix in cinema_pipeline.py — `f"amix=inputs={n_inputs}:duration={'longest' if use_standalone_dialogue else 'first'}:dropout_transition=2[aout]"`. Ship before Tier C if Tier C proceeds.
- **(b) Standalone fix**: separate `fix(audio):` commit. Same 1 LoC.
- **(c) NO ACTION**: 1.2s of silence at end of 5s clip is barely perceptible; Tier C reels with longer dialogue will mask the issue.

**Operator preferred:** **(a) Inline fold** if proceeding to Tier C — cheap fix, improves output quality.
