**IMPORTANT (I-A6.1): brief's `CINEMA_BUDGET_LIMIT_USD` env var is NOT
wired into `config/settings.py`.**

Investigation:
```
$ grep -n "budget\|BUDGET" config/settings.py
(no matches)

$ grep -n "budget" cinema/core.py
99:    budget_usd = settings.get("budget_limit_usd")
113:        cost_tracker=CostTracker(budget_usd=budget_usd),

$ grep "would_exceed\|is_over_budget" cost_tracker.py
328:        if self.budget_usd is None:
331:        return (self.spent_usd + cost) > self.budget_usd
338:        if self.budget_usd is None:
340:        return self.spent_usd > self.budget_usd
```

Conclusion: **`cost_tracker.py` budget enforcement is opt-in via
per-`CostTracker` instance `budget_usd` constructor arg**. When
`budget_usd is None` → `would_exceed()` and `is_over_budget()` both
return `False` (no enforcement, no warning). The `cinema/core.py`
construction path reads `settings.get("budget_limit_usd")` from a
**per-project settings dict**, NOT from `config/settings.py` env-var
loader.

The brief §3 A6 "Test project budget: review .env
CINEMA_BUDGET_LIMIT_USD" presumes an env-var binding that does not
exist in the global settings. Two paths to budget enforcement during
Tier B/C/D:

1. **Per-project settings.json** (per `domain/projects/<pid>/settings.json`):
   if the operator-created Tier B project has `budget_limit_usd: 50.0`
   set in its settings.json, enforcement activates for THAT project's
   pipeline. **Operator default for Tier B: explicit setting in
   project's settings.json before first generation call.**

2. **Wire `cinema_budget_limit_usd` into `config/settings.py`** and have
   `cinema/core.py` fall back to it when per-project setting is absent.
   This is a code change (~5 LoC); deferred to post-Tier-A as a
   `fix(settings):` or `feat(settings):` candidate if user-principal
   wants global ceiling enforcement.

**Disposition recommendation** per Rule #15 advisory matrix:
