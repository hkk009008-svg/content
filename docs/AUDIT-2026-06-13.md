## 0. Methodology & caveats (read first)

1. **The repo was live and self-mutating during the audit.** Five seats were committing to the shared tree in real time. `HEAD` advanced from `a10986c` → `f04efda` → `b060546` and the unpushed count went **4 → 10 → 12 ahead** across ~30 minutes (`git status -sb` sampled three times; `git reflog`). Every git/state observation here is a timestamped snapshot of a moving system, not a stable state.
2. **Audited from a Linux/Python-3.10 sandbox against a macOS repo.** The `.venv` is macOS Homebrew Python 3.13 (`.venv/bin/python → /opt/homebrew/...python3.13`), unusable here.
3. **The full §15 smoke (`scripts/ci_smoke.py`) could not be run by the auditor** — it imports the whole pipeline (`ModuleNotFoundError: No module named 'filelock'`, then heavier deps). Instead the two *lightweight* sub-gates it calls were run directly:
   - `scripts/check_coordination.py` → **ran clean (exit 0)**, advisories only.
   - `scripts/check_doc_claims.py` → **blocked** (`ModuleNotFoundError: No module named 'tomllib'`; needs Python ≥3.11). **Doc-anchor drift is therefore UNVERIFIED by this audit.**

## 2. Protocol coherence

### P1 — The routers describe 2 seats; 5 roles are live · **High**

`CLAUDE.md` and `AGENTS.md` both still state the operating model as two seats:

> "Two Claude sessions run in parallel by design — director-seat and operator-seat." (`CLAUDE.md` §Director–Operator; mirrored `AGENTS.md:205`)

Reality, by three independent signals:

```text
# presence heartbeats (one per live seat)
coordination/presence/{coordinator,director,director2,operator,operator2}.md
director2-heartbeat.ts, operator2-heartbeat.ts (+ director/operator)

# git log voices
"coord(mailbox): coordinator -> all ...", "director2 -> operator2 ...", "operator-1 WRAP" ...

# the routers never name the new roles
$ grep -rinc "director2|operator2|coordinator|four-seat" CLAUDE.md AGENTS.md
CLAUDE.md:0   AGENTS.md:0
```

`docs/protocol/claude/four-seat-extension.md` §7 explicitly prescribed the router edits this implies — *"Framing: 'two seats' → 'four seats / two pairs'"*, plus Rule #8/#19/#20 deltas — but those edits were **never applied to the routers**. A freshly-launched `director2`, `operator2`, or `coordinator` reading the canonical entry-point docs would find no description of its own seat, pair, lane, or the `all` broadcast target. This is the single biggest coherence gap: the load-bearing routers misrepresent the operating model.

### P2 — Rule #23 is operative but uncodified · **High**

Rule #23 ("lane ownership") governs current work — it appears in commits (`ef5c4c6` "Pair-A Rule#23 co-sign", brief `27d1323`), in the brief filename `BRIEF-director2-2026-06-13-landscape-char-routing-rule23-joint.md`, and across ~17 handoffs. Yet it is absent from every place a rule is supposed to live:

```bash
$ grep -rln "Rule #23|Rule#23|Rule 23" docs/protocol/  → ONLY four-seat-extension.md (lines 33,106,125)
# max rule actually in the governance file:
$ grep -ohE "Rule #?[0-9]+" docs/protocol/claude/director-operator.md | ... | tail -1  → 22
$ grep -ohE "Rule #?[0-9]+" docs/protocol/agents/director-operator.md  | ... | tail -1  → 22
# registry max:
$ grep -ohE "Rule #?[0-9]+" docs/PROTOCOL-RULES-LOG.md | ... | tail -1  → 22
# routers still claim the old range:
CLAUDE.md:193 / AGENTS.md:217 — "Full governance — Rules #7–#22 ... lives in director-operator.md"
```

So Rule #23 lives only in a **draft extension doc**, never folded into `director-operator.md` (the file the routers point to for "full governance"), never entered in `PROTOCOL-RULES-LOG.md` (the provenance registry that `R-EVIDENCE`/Rule #11 require), and the routers' "#7–#22" range is now stale. The repo's own codification discipline (Rule #11 codification-bias check; `R-MEASURE`/`R-EVIDENCE` "commit the instrument / cite the source") is being violated for its newest active rule. *Caveat: Rule #23 was minted today; the registry may simply be lagging by hours — but it is load-bearing now.*

### P3 — Cascading three-layer seat-model drift · Medium

Each documentation layer lags the next, so there is no single correct source:

```text
Layer 1  routers (CLAUDE.md/AGENTS.md)      →  2 seats
Layer 2  four-seat-extension.md             →  4 seats; §4 FROM enum = director|director2|operator|operator2  (no coordinator)
Layer 3  tooling (ground truth)             →  4 seats + `coordinator` (send-only) + `all` (broadcast target)
         coordination/bin/send-event:22     →  FROM: ...|coordinator
         scripts/check_coordination.py:44+  →  ROLES 4-tuple; frm-regex includes coordinator
```

Even the extension doc that fixed Layer 1 is itself stale vs the tooling it spawned (the `coordinator` pseudo-seat was added later, in the bin scripts + linter, but never back-ported into the doc). Confirmed live: `coordinator` authored 10 events in `mailbox/sent/` and has a `presence/coordinator.md` (no heartbeat file — consistent with a send-only meta-role).

### P4 — `four-seat-extension.md` status is self-contradictory · Medium

```text
line 1   : "# Four-seat coordination — protocol extension (DRAFT for director ACK)"
line 11  : "... Status: **ACCEPTED.**"
line 152 : "**Director ACK requested on:** (a) the seat IDs ... (b) the `all` broadcast target ..."
```

The body declares it ACCEPTED, TOOLING LANDED, PRINCIPAL-CONFIRMED, and the seats are demonstrably live — but the title still says DRAFT-for-ACK and the bottom still requests an ACK that has clearly already happened. A reader cannot tell whether this is a proposal or settled law. The draft scaffolding should be removed now that it shipped.

`coordination/README.md` does link the extension (good — a seat reading `coordination/` can find it), but the **routers and the CLAUDE/AGENTS migration-map do not**, and the migration-map predates the expansion entirely. The map that is supposed to explain the operative split is silent on the doubling of seats it now governs.

### P6 — "operator" overloaded in the precedence list · Low

`AGENTS.md:237-242` defines precedence as: *"1. The operator's explicit instructions (highest) → 2. CLAUDE.md extensions → 3. universal principles → 4. model default."* Here "operator" means the **human principal**, but everywhere else in the corpus "operator" is the **operator-seat AI** and the human is "user-principal." The one place precedence is pinned down is the one place the term flips meaning. Recommend "the user-principal's explicit instructions."

### P7 — agent-agnostic tree is missing the extension · Low

`docs/protocol/claude/` has `four-seat-extension.md` (156 lines); `docs/protocol/agents/` has no counterpart (`core / director-operator / failure-modes / orchestration` only). Low impact today because all live seats are Claude, but the "agent-agnostic root" (`AGENTS.md`) is no longer complete for a non-Claude tool.

## 3. Process hygiene

### H1 — Default git index stages deletion of in-HEAD tests · **High**

`git status --porcelain=v2` (run with no `GIT_INDEX_FILE`, i.e. the **default/shared** index) shows staged deletions (`D.`) of files that are confirmed present in `HEAD`:

```text
D.  tests/unit/test_lip_sync_best_of_failed.py     (IN HEAD)   ?? same path untracked on disk
D.  tests/unit/test_seedance_dispatch.py           (IN HEAD)   ?? same path untracked on disk
D.  tests/unit/test_stitch_modules.py              (IN HEAD)   ?? same path untracked on disk
D.  docs/BRIEF-director2-...rule23-joint.md         (IN HEAD)
D.  coordination/mailbox/sent/2026-06-13T12-00-27Z-coordinator-to-all-fyi.md   (IN HEAD)
D.  coordination/mailbox/sent/2026-06-13T12-03-19Z-coordinator-to-director2-... (IN HEAD)
D.  docs/HANDOFF-director-2026-06-13-PM6-...  /  docs/HANDOFF-operator-2026-06-13-PM6-...
MM  lip_sync.py        MM  phase_c_ffmpeg.py        (staged + further unstaged edits)
```

The deleted tests are exactly the ones backing the recent `D1`/`D4`/`D6` fixes in the last commits. **Why this is a live footgun, not cosmetic:** the dispatch protocol routes subagents to *this* index on purpose — `CLAUDE.md` / templates mandate subagents *"prefix all git with `env -u GIT_INDEX_FILE`"*, which **unsets** the per-seat index and falls back to the default one carrying these staged deletions. A subagent (or human, or hook) that commits without a tight pathspec would drop the tests/docs from the next commit; the untracked on-disk copies would *not* be auto-re-added. Most likely cause: the per-seat-`GIT_INDEX_FILE` design left the shared default index drifted after some `git rm`/re-add churn. **Recommended (report-only):** reset the default index to HEAD (`git read-tree HEAD` / `git reset` with no pathspec) once no commit is in flight, and confirm subagent commits always use explicit pathspecs.

### H2 — STATE.md claims "clean" while 41 files are dirty · Medium

Sampled at the same instant:

```bash
$ git status --porcelain | wc -l        → 41
STATE.md (auto-generated, @12:45:37Z):  - **Working tree:** clean
                                         - **Unread mailbox:** director=0, operator=0
$ python3 scripts/check_coordination.py → director:2  director2:5  operator:2  operator2:2  unread
```

Three problems: (a) the cleanliness flag is a **false "clean"** (the hook evaluates a per-seat index while the shared tree/default index is dirty) — a false "clean" is worse than "unknown"; (b) the unread counts contradict the linter; (c) it lists only `director`/`operator`, never `director2`/`operator2` — the `four-seat-extension.md §3` "cosmetic" hook-unread update was never applied. STATE.md is gitignored and self-labeled "stale cache / lowest authority," which bounds the blast radius, but it is the at-a-glance signal seats actually read. Recommend the hook emit `clean | DIRTY(n) | unknown` honestly and enumerate all four seats.

### H3 — Two overlapping status artifacts; STATUS.md abandoned · Medium

`STATE.md` (HEAD-move hook, near-live) and `STATUS.md` (`scripts/status.py --write`) overlap in purpose. `STATUS.md` was last generated **2026-05-29** and is wildly stale:

```text
STATUS.md:  sha cc8dec6 · "19 ahead" · latest ADR-017 · "anchor-drift: clean"
reality  :  HEAD b060546 · 12 ahead · ADR-025 in flight (git log)
```

Both are gitignored, so neither pollutes history, but a 2-week-old `STATUS.md` sitting next to a live `STATE.md` is a trap for any reader (human or seat) who opens the wrong one. Recommend retiring `STATUS.md` (or having the hook regenerate/delete it) and keeping a single status artifact.

### H4 — Shared-`.git` write contention · Medium

```bash
$ ls -la .git/index.lock   → -rw------- 0 bytes, created 12:45 (same minute as the HEAD-b060546 commit)
$ git status ...           → warning: unable to unlink '.../.git/index.lock': Operation not permitted
HEAD ahead count over the session: 4 → 10 → 12
```

Five seats share one working tree and one `.git`. Per-seat `GIT_INDEX_FILE` isolates *staging*, but the ref and `index.lock` are shared, so commits serialize on git's ref lock and a crashed/slow op can leave a lock that blocks peers. The 0-byte lock observed is likely an **in-flight** commit (transient) rather than a dead one, but with this concurrency it is a standing risk. Recommend a lock-staleness sweep in the start-of-turn hook (age-out + remove orphaned `index.lock`).

### H5 — Documentation sprawl / archive discipline not followed · Medium

```
ls docs/*.md | wc -l                       → 175 top-level markdown files
ls docs/ | grep -ciE '^HANDOFF'            → 121
ls docs/ | grep -oE '20..-..-..' | sort | uniq -c | tail -1  → 21 dated 2026-06-13
find docs/archive -type f | wc -l          → 2
```

The doc-map promises *"Past handoffs (historical) → docs/archive/"*, but 121 `HANDOFF-*` files accumulate at `docs/` root while `docs/archive/` holds two. At 21 handoffs in a single day, discoverability of the *canonical* docs (ARCHITECTURE, PROGRAM-MANUAL, the protocol set) is being drowned. Recommend an archival rotation (e.g. move handoffs older than N days, or all but the latest per seat/pair, into `docs/archive/<date>/`).

### H6 — Secrets hygiene is sound (positive finding) · Low

```
for f in .env token.pickle client_secrets.json; do git ls-files --error-unmatch $f; done → all "not tracked"
git log --oneline --all -- .env token.pickle client_secrets.json                          → (empty: never committed)
```

`.gitignore` is comprehensive and correct (env, OAuth secrets, `*-key.json`, `*.pem`, media, runtime trees, per-seat launch config, presence, STATE/STATUS). The only residual notes: (a) `.env` holds **23 live API keys** in plaintext in this synced/mounted folder — fine for version control, but an operational exposure if the folder itself is ever shared (worth a mental note given this very Cowork mount can read it); (b) `.claude/settings.local.json` is untracked and not gitignored (trivial — add a rule).

### H7 — Truth-layer freshness mechanism lapsed · Info

```bash
grep -n "Last verified" ARCHITECTURE.md     → one line (1854): "2026-05-24"
git log --oneline --since=2026-05-24 | wc -l → 1281
```

`ARCHITECTURE.md` §18 prescribes keeping the file fresh, and `R-START` says re-verify any section a commit has touched since its `Last verified` footer. With a single 3-week-old stamp over the whole 1868-line doc and 1281 commits since, effectively the entire truth layer is past its verification horizon. *Line-by-line truth-drift was out of audit scope, and the repo's own anchor-drift gate (`check_doc_claims.py`) could not be run here (needs Python ≥3.11) — so the actual drift magnitude is **unverified**; this is a finding about the discipline, not a count of stale claims.*

## 4. What's working (so the audit is balanced)

The protocol is unusually rigorous and much of it is healthy. The mailbox **passes its own linter clean** (`check_coordination.py` exit 0; only a benign `cursor_orphan` advisory on `operator` + INFO unread counts). Every link in the `CLAUDE.md` doc-map **resolves** (14/14 targets present, including `PROGRAM-MANUAL.md`, `archive/`, the protocol set). Secrets are clean (H6). The CLAUDE↔AGENTS split is well-conceived — `AGENTS.md` is the agent-agnostic root, `CLAUDE.md` the Claude-mechanics companion, with an explicit precedence table. The per-seat `GIT_INDEX_FILE` design is a sound answer to multi-seat staging, and the tooling (`send-event`/`consume-events`/`check_coordination`) is internally consistent and enum-guarded. The `R-EVIDENCE`/`R-MEASURE` culture (cite the command, commit the instrument) is exactly the right backbone — the findings above are mostly cases where that culture has *not yet caught up with* the 2→4→5-seat expansion that landed in a single day.

## 5. Recommended fixes, prioritized (report-only — none applied)

1. **(P1/P2/P3) Reconcile the seat model top-down in one change.** Update `CLAUDE.md` + `AGENTS.md` framing to 4 seats / 2 pairs + coordinator; fold Rule #23 into `director-operator.md` (both `claude/` and `agents/`); add the Rule #23 entry to `PROTOCOL-RULES-LOG.md`; fix the "#7–#22" range to "#7–#23"; back-port `coordinator` into `four-seat-extension.md §4`.
2. **(H1) Clean the default git index** to HEAD when no commit is in flight, and add a guard that subagent commits use explicit pathspecs (since `env -u GIT_INDEX_FILE` routes them to that index).
3. **(P4/P5) De-draft `four-seat-extension.md`** (remove DRAFT title + ACK-request section) and link it from the routers + migration-map so it is discoverable.
4. **(H2/H3) Fix the status artifacts:** make the STATE.md hook report honest cleanliness + all four seats; retire or auto-regenerate the stale `STATUS.md`.
5. **(H5) Institute handoff archival rotation** into `docs/archive/`.
6. **(H4) Add an `index.lock` staleness sweep** to the session-start hook.
7. **(H7) Re-establish per-section `Last verified` footers** in `ARCHITECTURE.md`, and run `check_doc_claims.py` (on a Python ≥3.11 host) to quantify current anchor drift — which this audit could not measure.
8. **(P6/H6-b) Trivia:** disambiguate "operator"→"user-principal" in the precedence list; gitignore `.claude/settings.local.json`.
